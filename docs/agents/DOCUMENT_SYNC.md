# Document Sync

## Policy

Document Sync is explicit-only. Do not update agent docs unless the user asks for Document Sync, docs sync, agent docs update, architecture/runtime docs update, changelog update, or equivalent wording.

## Source-of-Truth Priority

1. Current stable implementation.
2. Runtime behavior, tests, generated artifacts, and build output.
3. Current agent docs.
4. Changelog and decision records.
5. Task notes.
6. External references.

Task notes and unfinished work are not implementation truth.

## Sync Workflow

1. Identify the requested baseline.
2. Read `AGENTS.md`.
3. Read this file.
4. Inspect code before editing docs.
5. Compare implementation to docs.
6. Classify the impact.
7. Update only affected docs.
8. Keep incomplete work in `docs/agents/tasks/`.
9. Add changelog entries only for meaningful documented changes.

## Impact Levels

Classify every Document Sync request by the highest verified impact level.

- Framework-level: architecture, runtime baseline, release policy, package topology, cross-package behavior, or framework-wide dependency/runtime changes.
- Package-family-level: changes that affect a whole family such as `common`, `databases`, `brokers`, `adapters`, `integrations`, `tools`, or `apps`.
- Package-level: changes isolated to one package, app, adapter, broker, database, integration, or tool.
- No-doc-impact: formatting, local cleanup, test-only edits, minor internal implementation details, or dependency noise with no runtime or public contract impact.

## Documentation Routing

- Framework-level changes may update `ARCHITECTURE.md`, `RUNTIME.md`, `PROJECT_CONTEXT.md`, `REPO_MAP.md`, `RULES.md`, `DECISIONS.md`, and `docs/agents/CHANGELOG.md`.
- Package-family-level changes may update the nearest package-family `AGENTS.md`, package-family `README.md`, `REPO_MAP.md`, or `SEARCH_HINTS.md` when ownership or navigation changed.
- Package-level changes may update the package `README.md`, focused package `AGENTS.md` when present, and package notes only when public behavior, API, config, or runtime semantics changed.
- Agent documentation workflow changes update `docs/agents/*` and `docs/agents/CHANGELOG.md`.
- Package `CHANGELOG.md` files are generated by the release workflow and should not be manually edited during normal Document Sync unless correcting release documentation before publish.

## Changelog Rules

`docs/agents/CHANGELOG.md` records agent-facing history for meaningful framework, architecture, runtime, repository policy, and Agent Docs changes.

Each package `CHANGELOG.md` records package release notes generated from conventional commits during release.

Future entries must describe implementation-backed changes only.

Do not add changelog entries for plans, experiments, formatting-only edits, or unpublished incomplete work.

Do not add agent changelog entries for package-local changes unless the change affects architecture, runtime behavior, public contracts, release policy, or future-agent navigation.

## In-Progress Work Handling

If unfinished work exists:

- exclude it from architecture, runtime, contracts, and project context
- record it only as technical debt or task material
- mark uncertainty explicitly when code intent is not complete

## Generated and Contract Docs

Contract docs must describe implemented controllers, message patterns, public interfaces, and generated schema behavior. If generated artifacts are stale, say so in task notes rather than changing implementation truth.
